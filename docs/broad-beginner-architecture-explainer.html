00

You do not need to understand everything at once.

A real system is not one clever file. It is a collection of small agreements: what data means, who owns it, how it moves, where it waits, and what happens when something fails.

Islandflow watches market activity. Raw trades, quotes, options prints, and news arrive from provider adapters. The system turns those observations into typed events, moves them through a message bus, stores them, derives higher-level facts, and sends useful views to a browser. The browser is the last stop, not the whole product.

The goal of this guide is transfer. You should leave able to look at another TypeScript system and ask better questions: Where does data enter? What validates it? Is this storage durable or temporary? Which process owns this decision? How does the UI receive changes?

FIRST ANALOGY

Picture a busy restaurant. Adapters are delivery doors, NATS is the ticket rail, services are specialized stations, ClickHouse is the permanent order archive, Redis is the whiteboard of what is hot right now, the API is the expediter, and React is the board the customer actually sees. The analogy is imperfect—but it gives every moving part a job.

01

The repository is a neighborhood, not a single house.

Islandflow is a monorepo: several applications, background services, and shared libraries live in one Git repository and use one Bun workspace.

What “Bun + TypeScript monorepo” actually means

Bun runs JavaScript/TypeScript, installs dependencies, and runs tests and scripts. TypeScript is JavaScript with a type system that lets the code describe expected data shapes. A monorepo keeps related projects together so they can share types and tools without publishing a package to the internet after every edit.

islandflow/ ├── apps/ │ ├── web/ Next.js + React interface │ └── desktop/ thin Electron wrapper ├── services/ │ ├── ingest-options/ options trades and quotes enter here │ ├── ingest-equities/ stock trades and quotes enter here │ ├── ingest-news/ news enters here │ ├── compute/ turns raw events into derived evidence │ ├── candles/ builds OHLCV candles │ ├── api/ HTTP + WebSocket boundary for clients │ └── replay/ re-publishes old events in order ├── packages/ │ ├── types/ shared event vocabulary │ ├── bus/ NATS subjects and helpers │ ├── storage/ ClickHouse queries and tables │ ├── config/ environment parsing │ └── observability/ logs and metrics └── deployment/ how the pieces run together

Why split code this way?

A folder boundary should answer an ownership question. A service owns a running job. A package owns reusable knowledge. An app owns a user-facing program. This is a broadly useful distinction: if code performs a continuous task, it may belong in a service; if it defines a shared contract, it likely belongs in a package.

Service

A program with a lifecycle: start, connect, work, handle errors, shut down. Example: services/candles.

Package

Code imported by other code. It normally does not run by itself. Example: packages/types.

A useful reading trick

When a large file feels opaque, do not read top-to-bottom. First scan its imports, environment schema, startup function, subscriptions, and shutdown function. Those five landmarks usually reveal what the program depends on, what it owns, and how it spends its time.

02

Events are the spine of the system.

Instead of one giant function calling everything else, Islandflow records small, explicit statements about what happened. Other parts of the system react to them.

An event is data describing a fact at a point in time: “a stock traded,” “an options quote changed,” or “a one-minute candle closed.” It is usually plain JSON, which makes it portable between processes and easy to store.

SEALED POSTCARD

Think of an event as a postcard. The sender writes a fact, stamps it with identity and time, and drops it into the system. Receivers should not secretly edit that postcard. If they learn something new, they create a new postcard that points back to the evidence.

packages/types/src/events.ts · simplified
const EquityPrintSchema = EventMetaSchema.merge(
  z.object({
    ts: z.number().int().nonnegative(),
    underlying_id: z.string().min(1),
    price: z.number().nonnegative(),
    size: z.number().int().positive(),
    exchange: z.string().min(1),
    offExchangeFlag: z.boolean()
  })
);

type EquityPrint = z.infer<typeof EquityPrintSchema>;
Schema
A runtime guard. It checks real incoming values.
Type
A compile-time guide. Your editor and compiler use it while you code.
z.infer
Build the TypeScript type from the Zod schema, avoiding two copies of the contract.

Portable idea: a type is a promise, validation is evidence

TypeScript types disappear when code runs. External data can still be malformed. That is why Islandflow uses Zod schemas such as EquityPrintSchema.parse(candidate) at boundaries. Static types help developers; runtime schemas protect the running system.

03

Follow one equity trade from wire to screen.

Click each step. The point is not to memorize file names; it is to see the same few responsibilities repeat in a predictable order.

1. A provider adapter receives a trade

The selected adapter—synthetic for safe local work or Alpaca for a provider connection—translates provider-specific data into Islandflow's common EquityPrint shape. Everything downstream can ignore the provider's peculiar field names.

services/ingest-equities/src/adapters/* → onTrade(candidate)

Developer question: Where is provider-specific mess converted into domain language?

The key decoupling move

The ingest service does not call the candle service directly. It publishes to a subject. Candles, compute, and API consumers can listen independently. That means a new consumer can be added without teaching ingest about it, and one worker can be restarted without turning the whole application into a single process.

Ingestpublishes equities.prints
Candlesupdates OHLCV windows
Computejoins quotes and infers signals
APIupdates live client state

This pattern is called publish/subscribe. One publisher states a fact; many subscribers choose whether that fact concerns them.

04

Each service is a worker with a narrow job.

“Microservice” can sound grand. Here it mostly means a separately running program with its own inputs, outputs, configuration, and shutdown behavior.

ingest-options

Normalizes option prints and NBBO quotes, enriches them with quote context, persists them, and publishes raw/signal streams.

Reads: synthetic, Alpaca, Databento, IBKR; equity quotes.
Writes: ClickHouse + option NATS subjects.

ingest-equities

Normalizes stock trades and quotes from synthetic or Alpaca adapters.

Reads: provider adapter.
Writes: ClickHouse + equities.prints/equities.quotes.

ingest-news

Backfills and listens for market news, converts it into canonical NewsStory events, and publishes them.

Reads: Alpaca news paths.
Writes: NATS; API persists consumed stories.

compute

The interpretation workshop: clusters option activity, builds flow packets, scores smart-flow hypotheses, joins equity prints to quotes, and infers dark/off-exchange evidence.

Reads: option/equity subjects, ClickHouse context, Redis rolling state.
Writes: derived ClickHouse tables + derived NATS subjects.

candles

Groups individual equity prints into time windows and calculates open, high, low, close, volume, and trade count.

Reads: equities.prints.
Writes: ClickHouse, NATS, and a Redis sorted-set cache.

api

The client boundary. It offers REST history/current reads, WebSocket live feeds, detail hydration, rate limiting, CORS, and synthetic controls.

Reads: NATS, ClickHouse, Redis, HTTP requests.
Writes: HTTP/WS responses; live Redis state.

replay

Reads saved or generated events, orders them, waits according to replay speed, and republishes them as if they were arriving live.

Reads: ClickHouse or synthetic fixture.
Writes: the same raw NATS subjects used by live ingest.

refdata / eod-enricher

Supporting and future-facing workers for reference/event-calendar knowledge and end-of-day enrichment.

Status: scaffolding and incomplete paths; useful reminders that architecture can include planned seams without pretending they are finished.

Services are not automatically “better”

Separate processes cost more: network connections, retries, ordering rules, deployment, monitoring, and partial failure. The split earns its keep when jobs truly have different lifecycles, scaling needs, or ownership. A good developer learns to see both the boundary's benefit and its tax.

05

NATS moves, ClickHouse remembers, Redis keeps the hot answer close.

These tools overlap just enough to be confusing. The cleanest way to learn them is by the question each one answers.

Question: who needs to hear that something happened?

NATS is the communication layer. JetStream adds stored streams, durable consumers, delivery policies, and acknowledgements.

A publisher sends JSON to a named subject, such as equities.prints. JetStream keeps messages for a configured time/size window. A durable consumer remembers its position and explicitly acknowledges processed messages.

Not the permanent analytical database: JetStream retention is deliberately bounded. It supports reliable movement and catch-up, while ClickHouse owns long-lived queryable history.

Question: what happened over a large span of time?

ClickHouse is a column-oriented analytical database. It is good at scanning many rows while reading only the columns a query needs—useful for time-series market data.

Ingest services persist raw facts. Compute and candles persist derived facts. The API asks for recent windows, cursor pages, ranges, replay slices, and evidence lookups.

Durable does not mean magical: table design, ordering keys, retention, disk space, and duplicate behavior still matter. The packages/storage package centralizes those details.

Question: what answer is needed repeatedly, right now?

Redis is an in-memory data structure server. Islandflow uses it as a hot cache and rolling-state surface, not as the canonical historical record.

The API keeps hot live channel snapshots and cursors there. Candles store recent time-ordered values in sorted sets. Compute can flush rolling statistics so expensive context is close at hand.

Redis can disappear: robust code can hydrate live state from ClickHouse and continue with reduced performance when the cache is unavailable.

The durable + hot pattern travels well

Many systems keep a slower source of truth and a faster disposable view. You will see this in ecommerce catalogs, social feeds, analytics dashboards, and game leaderboards. The architectural question is always: can the fast view be rebuilt from durable truth?

06

Read the TypeScript as sentences.

Syntax becomes less intimidating when you translate it into actors, actions, and constraints. Here are the recurring patterns in Islandflow.

Imports say “this file does not know everything”

services/ingest-equities/src/index.ts · representative
import { publishJson, SUBJECT_EQUITY_PRINTS } from "@islandflow/bus";
import { insertEquityPrint } from "@islandflow/storage";
import { EquityPrintSchema } from "@islandflow/types";
  1. @islandflow/bus owns how messages move and what subjects are named.
  2. @islandflow/storage owns database details.
  3. @islandflow/types owns what an equity print means.

async / await means “this takes time”

the ingest boundary · simplified from live code
onTrade: async (candidate: EquityPrint) => {
  const print = EquityPrintSchema.parse(candidate);

  await insertEquityPrint(clickhouse, print);
  await publishJson(js, SUBJECT_EQUITY_PRINTS, print);
}

Read it aloud: “When a trade arrives, validate it. Wait for ClickHouse to store it. Then wait for JetStream to publish it.” The ordering is intentional. A network call returns a Promise; await pauses this function without freezing the entire runtime.

Generics describe a slot without choosing its contents yet

packages/bus/src/jetstream.ts · real helper
export const publishJson = async <T>(
  js: JetStreamClient,
  subject: string,
  payload: T
): Promise<void> => {
  const codec = JSONCodec<T>();
  await js.publish(subject, codec.encode(payload));
};

T means “whatever payload type the caller supplies.” The same helper can publish an EquityPrint, NewsStory, or FlowPacket without using the unsafe catch-all type any.

for await reads a stream over time

durable consumer loop · simplified
for await (const msg of subscription.messages) {
  const event = Schema.parse(subscription.decode(msg));
  await handleEvent(event);
  msg.ack();
}

This loop does not race through a finished array. It waits for each new message. The explicit ack() tells JetStream the consumer finished. That creates at-least-once delivery: after a crash, the system may deliver an unacknowledged message again, so handlers should be prepared for duplicates.

07

The API translates system truth into client-friendly views.

The web app does not connect directly to ClickHouse, Redis, or NATS. The API owns that boundary and exposes HTTP and WebSocket contracts.

Two ways the browser asks for data

HTTP / REST

The browser asks a question and receives one response. Good for initial snapshots, history pages, ranges, and detail lookups.

GET /history/equities?before_ts=…

WebSocket

The connection stays open. Server and client can exchange messages over time. Good for live prints, quotes, candles, alerts, and health.

new WebSocket("/ws/live")

React: UI as a function of state

React components return a description of the interface. When state changes, React calculates what the browser DOM needs to update. Islandflow’s terminal code uses hooks heavily because live data, filters, focus, history, and replay all affect the visible result.

apps/web/features/terminal/state.tsx · simplified

  () => parseTickerFilterInput(filterInput),
  [filterInput]
);

const liveSession = useLiveSession(
  mode === "live",

);
  1. useState stores a value that can change and trigger a render.
  2. useMemo derives a value and recalculates it only when dependencies change.
  3. A custom hook such as useLiveSession packages reusable behavior behind a React-friendly interface.

Where beginners often get tangled

React state is not the database. It is the browser's current working model. The API cache is not the database either. Neither is Redis. Asking “which layer owns the durable truth?” will prevent a surprising number of bugs.

Server-composed rows reduce frontend detective work

A raw option print is not always enough for a useful row. The API’s durable-row code can combine prints, NBBO, flow packets, and smart-flow support into a client-ready view model. This is a tradeoff: more server work, but fewer fragile joins duplicated in the browser.

08

Live, history, and replay are three views of time.

They can share event contracts while using different delivery mechanics. That separation is central to event-sourced systems.

Live

“What is happening now?” The API consumes NATS subjects, maintains bounded hot state, and fans new events to subscribed WebSockets.

Optimizes freshness, connection health, bounded memory, and stable UI updates.

History

“What happened before this point?” The API queries ClickHouse using cursor pairs such as timestamp + sequence.

Optimizes deterministic pagination and large retained datasets.

Replay

“What if old events arrived again, in their original order?” The replay service merges streams and republishes them at a chosen speed.

Optimizes repeatable investigation, demos, and testing of downstream behavior.

Why timestamp + sequence?

Millisecond timestamps can tie. Pagination using only “before this timestamp” may skip or repeat rows. A cursor with ts and seq creates a stable total order: first compare time, then use sequence to break ties.

PHOTO FINISH

Two runners cross during the same displayed millisecond. The timestamp says it was a tie; the sequence is the photo-finish ordering. A stable cursor needs both.

Replay is possible because events carry their own meaning

If downstream logic depends only on “whatever happens to be in process memory,” old data is hard to reproduce. Islandflow’s direction is to make inputs explicit enough that the same compute pipeline can process a live event or a republished historical event.

Event sourcing is a direction, not a purity badge

Islandflow still has caches, mutable in-process maps, materialized tables, and composed views. “Event-sourced” does not mean every byte is reconstructed from an eternal log on every request. It means important facts are explicit and replayable, while derived state can be rebuilt or explained from them.

A “real developer” thinks about the unhappy path.

The jump from toy programs to production systems is less about fancy syntax and more about recognizing time, failure, ownership, and uncertainty.

Notice the vocabulary in the code

Names such as connectJetStreamWithRetry, buildDurableConsumer, LIVE_FRESHNESS_THRESHOLDS, shuttingDown, and fetchRecent… are architecture made visible. Good names let you read the safety model before understanding every implementation detail.

Five questions to carry into any codebase

  1. What is the source of truth?
  2. What can be rebuilt, and from what?
  3. What happens if this operation runs twice?
  4. Which clock or ordering rule does this depend on?
  5. How does failure become visible to a human or another service?

Observability is how the system tells its story

Logs describe individual events and errors. Metrics count patterns over time. Health endpoints answer a narrow availability question. A trace ID connects related facts. None replaces the others. When you add behavior, ask what evidence would let you prove it worked—or explain why it did not.

10

Learn by following one narrow thread end to end.

Do not start by “learning all of React” or “understanding the whole backend.” Build a chain of small, evidenced wins.

A one-session route

Use synthetic data. Keep the scope to equity prints so every hop has an obvious name.

  1. Open packages/types/src/events.ts. Write down the fields of EquityPrint in your own words.
  2. Open services/ingest-equities/src/index.ts. Find validation, ClickHouse insertion, and NATS publication.
  3. Open packages/bus/src/subjects.ts. Find equities.prints.
  4. Open services/candles/src/index.ts. Find the subscription and emitCandle.
  5. Open services/api/src/index.ts. Find the same subject and a history route.
  6. Open apps/web/features/terminal/live.ts. Find fetch and new WebSocket.
  7. Draw the path from memory. Then compare it to chapter 03.

Small experiments that teach real skills

  • TypeScript: create a tiny object missing exchange and see what the type checker says.
  • Zod: call EquityPrintSchema.safeParse with bad data and inspect the error structure.
  • Async: add timing logs around an awaited function and observe ordering.
  • React: build a 10-line component with useState and a filter input before reading Islandflow’s larger state module.
  • Events: write a 20-line in-memory publisher with two subscribers to feel fan-out directly.
  • Testing: change one expected schema field in a focused test, watch it fail, restore it, and watch it pass.

You do not need to recall every API. Progress is being able to predict where a concern belongs, form a testable hypothesis, use tooling to find the exact code, and explain what evidence would confirm your change. That is already real development.

11

A pocket glossary and source map.

Architecture vocabulary is compressed meaning. Use these definitions as handles, then return to the code whenever a word starts feeling abstract.

Adapter
Translates an external provider or protocol into the application's own interface and vocabulary.
API
An agreed way for software components to communicate; here, especially HTTP and WebSocket endpoints.
Cache
A faster copy or derived view that should be safe to rebuild from a source of truth.
Consumer
A process that receives and handles messages from a stream or subject.
Cursor
A stable position used to request the next or previous page of ordered data.
Derived event
A new fact calculated from earlier facts, such as a candle, join, packet, or alert.
Durable
Expected to survive process restarts; always ask what storage and retention policy make that true.
Event
An immutable description of something that happened, with identity and time.
Event sourcing
An architecture that treats recorded events as the basis for rebuilding or explaining state.
Hydration
Filling a partial view with data from storage or another endpoint.
Idempotent
Safe to perform more than once without changing the final result after the first success.
NBBO
National Best Bid and Offer: the best displayed buy and sell prices across venues.
OHLCV
Open, high, low, close, and volume—a compact summary of trades in a time window.
Publisher
A component that sends a message without needing to know every receiver.
Schema
An executable description that validates the shape and rules of runtime data.
Service
A separately running program that owns a continuous job and lifecycle.
Subject
A named NATS channel to which publishers send and consumers subscribe.
WebSocket
A long-lived, two-way connection between browser and server.
