Islandflow implementation turn

Smart-flow alerts phase 03 added canonical derived alert contracts and delivery surfaces.

Beads islandflow-ghce.3 Branch lavender/smart-flow-alerts-03-derived-alerts 2026-06-26

Summary

Added SmartFlowAlertEvent as the canonical alert contract derived from non-abstained native smart-flow projections. The new alert path has its own JetStream subject and stream, ClickHouse table, live-cache channel, REST history and replay routes, and websocket route. Legacy alert and classifier-hit surfaces remain untouched, and the web terminal only gained passive live-session support for the new channel rather than switching any UI consumer. Reviewer hardening tightened projection-derived invariants, fixed same-cursor alert paging and live-cache identity, and moved compute alert validation ahead of publish side effects.

Implementation

Canonical contract

SmartFlowAlertEvent carries event metadata, schema version, hypothesis refs, policy confidence, evidence quality, a non_abstained_hypothesis trigger, the source projection, and evidence refs.

Projection-only derivation

Compute derives alerts only through smartFlowAlertFromProjection after a native smart-flow projection is emitted, and returns null for abstentions or compatibility rows.

Dedicated delivery

Added flow.smart_flow_alerts on SMART_FLOW_ALERTS, plus smart-flow-alerts live, history, replay, flow, and websocket surfaces.

Storage and cache

Added smart_flow_alerts ClickHouse storage with JSON round-trip parsing, alert-id tie ordering, boundary-tie paging, and live-cache hydration support.

Reviewer hardening

Canonical alerts now reject nested abstention or compatibility drift, preserve same-cursor alert siblings, and validate the whole compute alert plan before any publish loop side effects.

+export const SMART_FLOW_ALERT_TRIGGER_KIND = "non_abstained_hypothesis";
+
+export const isNativeNonAbstainedSmartFlowProjection = (
+  projection: SmartFlowExplainabilityProjection
+): boolean =>
+  projection.source_channel === "smart-flow" &&
+  projection.hypothesis.generated_from === "flow_evidence_cluster" &&
+  projection.abstention.abstained === false &&
+  projection.hypothesis.abstention.abstained === false &&
+  projection.insight.abstention.abstained === false &&
+  projection.compatibility === undefined &&
+  projection.hypothesis.compatibility === undefined &&
+  projection.insight.compatibility === undefined;

+export const SmartFlowAlertEventSchema = EventMetaSchema.extend({
+  schema_version: SmartFlowContractVersionSchema,
+  alert_id: z.string().min(1),
+  hypothesis_id: z.string().min(1),
+  insight_id: z.string().min(1),
+  underlying_id: z.string().min(1),
+  hypothesis_type: FlowHypothesisTypeSchema,
+  direction: SmartMoneyDirectionSchema,
+  policy_confidence: z.number().min(0).max(1),
+  evidence_quality: z.number().min(0).max(1),
+  trigger: SmartFlowAlertTriggerSchema,
+  projection: NonAbstainedNativeSmartFlowProjectionSchema,
+  evidence_refs: z.array(z.string().min(1)).min(1)
+})
+  .strict()
+  .superRefine((alert, ctx) => {
+    // Alert metadata must stay projection-derived, not independently mutable.
+  });
+
+export const smartFlowAlertFromProjection = (
+  projection: SmartFlowExplainabilityProjection,
+  options: { alert_id?: string; trace_id?: string } = {}
+): SmartFlowAlertEvent | null => {
+  if (!isNativeNonAbstainedSmartFlowProjection(projection)) {
+    return null;
+  }
+
+  const alertId = options.alert_id ?? `smartflow:alert:${projection.refs.hypothesis_id}`;
+  return SmartFlowAlertEventSchema.parse({
+    source_ts: projection.source_ts,
+    ingest_ts: projection.ingest_ts,
+    seq: projection.seq,
+    trace_id: options.trace_id ?? alertId,
+    schema_version: projection.schema_version,
+    alert_id: alertId,
+    hypothesis_id: projection.refs.hypothesis_id,
+    insight_id: projection.refs.insight_id,
+    underlying_id: projection.hypothesis.underlying_id,
+    hypothesis_type: projection.hypothesis.hypothesis_type,
+    direction: projection.hypothesis.direction,
+    policy_confidence: projection.hypothesis.scores.confidence.policy_confidence,
+    evidence_quality: projection.hypothesis.scores.confidence.evidence_quality,
+    trigger: {
+      kind: SMART_FLOW_ALERT_TRIGGER_KIND,
+      projection_trace_id: projection.trace_id,
+      projection_version: projection.projection_version,
+      source_channel: projection.source_channel
+    },
+    projection,
+    evidence_refs: projection.refs.evidence_refs
+  });
+};

Risks

  • Deployment needs the new SMART_FLOW_ALERTS stream and smart_flow_alerts table reconciled before canonical alerts appear through live APIs.
  • Alert priority remains a UI concern for phase 04; no stored legacy score, severity, or classifier hits were added to the canonical event.
  • Smart-flow alert history and replay may return extra rows that share a page-boundary cursor; this preserves correctness with the existing two-field cursor shape.
  • Old alert routes intentionally remain available until the later legacy-removal phase, so both alert surfaces can coexist during migration.
