Summary

Implemented synthetic market-data phase 03 as one reviewable batch. The package now has a deterministic scenario catalog, separate ground-truth labels, and smart-flow expected-output sidecars for positive alert, no-alert, suppression, evidence requirement, forbidden evidence, and false-positive validation.

Changes Made

  • Added packages/synthetic-market/src/scenarios.ts with ScenarioInjection, GroundTruthLabel, evidence requirement, expected output, and scenario fixture artifact contracts.
  • Defined named deterministic scenarios for institutional directional flow, retail attention flow, event/noise positioning, volatility seller behavior, hedge-reactive flow, structure arbitrage, and a no-alert negative case.
  • Extended fixture manifests so scenario fixtures can point to separate labels.json and smart-flow-expected-outputs.json artifacts with stable hashes and counts.
  • Added scenario fixture builders and writers that generate canonical market events plus sidecar label and expected-output files.
  • Exported @islandflow/synthetic-market/scenarios as a package subpath.
  • Added tests for catalog coverage, deterministic sidecar generation, no label leakage into market event payloads, no-alert expectations, trace-id validity, and infra-free sidecar writing/loading.

Context

The active phase document is docs/implementation/synthetic-market-data/03-scenarios-labels-expected-outputs.md. It asks for named deterministic scenarios, labels that remain separate from emitted market events, and expected-output manifests that downstream smart-flow scoring can use for positive, negative, abstention, and false-positive validation.

This batch intentionally did not add UI controls, live service behavior, replay integration, historical calibration, or smart-flow scoring changes. It only creates deterministic fixture contracts and sidecar expectations.

Important Implementation Details

  • market-events.json remains canonical event-only output. Scenario IDs and labels are kept out of emitted market event payloads and are verified by tests.
  • Scenario fixtures write sidecars through labels_path and smart_flow_outputs_path. The main manifest records hashes and counts so reviewers can audit deterministic output without loading a service.
  • Every label is keyed by run_id, scenario_id, trace/event refs, expected class, expected direction, confidence band, required evidence, forbidden evidence, and a false-positive penalty.
  • Expected outputs use smart-flow vocabulary such as directional_accumulation, retail_attention_flow, event_positioning, volatility_supply, structure_arbitrage, hedge_rebalance, and unclear.
  • The no-alert fixture explicitly expects no flow_hypothesis_event and no smart_flow_insight, with a high false-positive penalty.

Expected Impact for End-Users

There is no direct UI or live-service change. The end-user impact is upstream: future smart-flow scoring and replay work now has deterministic, named, reviewable fixtures that include both should-detect and should-not-alert cases.
