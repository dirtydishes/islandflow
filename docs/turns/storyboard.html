How Islandflow turned from a terminal into an evidence pipeline.

This page reads the turn documents, implementation phase folders, and Beads epics as one story: the UI got quieter, the deploy path got stricter, terminal.tsx was broken into focused modules, synthetic and smart-flow gave the product explainable evidence, durable tapes became reusable, and the public edge was narrowed around the hosted UI.

The Story Arc

The turn docs describe a project where I kept discovering where the source of truth needed to move. At first the work was mostly about making the terminal usable. Then the deployment path became the thing to stabilize. Then Beads epics and implementation phase docs became my operating board for multi-PR streams. Only after that did the product surface absorb synthetic data, smart-flow, terminal extraction, charting, durable tape modules, performance budgets, and private-edge hardening.

Opening move

May 14 to May 15

The terminal got hardened before it got ambitious.

First, I decided the terminal had to behave like an operator surface instead of a fragile demo. The early turns tightened responsiveness, keyboard behavior, constrained-width layout, and visual intensity. Then, after that hardening branch met main, conflict work preserved both the new terminal coverage and the ticker parsing already present on main.

The important decision was restraint: quiet the chrome, keep the dark evidence-console identity, and let live data carry the weight. That choice becomes the visual baseline for the rest of the folder.

First twist

May 15 to May 19

The product could not move faster than the deploy path.

Then, the deployment story took over. Docker was declared the default VPS path, but the repo still needed native units, fast deploy modes, workspace snapshot checks, remote resolution that did not assume origin, and warnings for stale compose stacks. Each fix exposed another edge: frozen lockfile drift, missing history routes, NPM routing, systemd readiness, and remote precheck assumptions.

Then came the bigger infrastructure fork: native public edge cutover. The docs show it as a practical detour, not a rewrite for its own sake. I added units, fallback scripts, firewall and proxy adjustments, and a safer path for worker-only iteration. Then the news ingest service forced another repair, because native templates, Alpaca auth, backfill limits, and websocket fanout all had to agree.

dual runtime docker preflight docker speed fast mode native edge native news repair

Data persistence

May 16 to May 20

The terminal stopped depending on the live cache.

While deploy work was being tightened, the tape work found a product truth: operators scroll away from the live head, and the interface cannot punish them for investigating history. The tape began to hold when scrolled, resume at the top, and lazy-load ClickHouse history. After that, the same principle reached alert drawers, where selecting an alert had to hydrate durable context from ClickHouse instead of hoping linked evidence still lived in memory.

Then a subtle bug made the lesson sharper. A historical alert looked broken because the UI assumed the first evidence reference was always a flow packet. Compute alerts often pointed first to a smart-money event. Fixing that made the drawer trust all evidence refs, rather than only the first one.

Workflow correction

May 17 to May 30

The repo changed how work is remembered, reviewed, and published.

Then, project memory moved out of informal notes and into durable systems. Beads got a Dolt remote, Forgejo became canonical, docs publishing produced an index, and turn-document rules were clarified so future implementation work would leave a reviewable trail. The repo also backfilled remotes between GitHub and Forgejo, then made Forgejo CI the baseline instead of a side channel.

After that, the quality gates expanded. Typecheck became a first-class command, CI started running formatting, lint, type safety, tests, route checks, Docker snapshot drift checks, and the production web build. Then the remaining CI failures became concrete repairs: dependency resolution, terminal test mocks, and vulnerable package overrides.

Product pivot

May 23 to June 13

The UI stopped being a home page and became route-specific instrumentation.

Then, the surface changed shape. Tape became Options, the web rail moved into a top-left drawer, and the home page became a command deck. The mock route sequence started as exploration, but it became the decisive design process: first four different dashboard directions, then four no-card routes, then four dense operational studies.

The turn docs show the selection pressure clearly. I refined the command deck header, rejected generic card composition, and eventually rebuilt production Dashboard, Options, and News around mock9 through mock12. Then local development had to catch up: hosted API defaults, CORS preflight, desktop workflows, and terminal evidence text all had to support the new hosted shape.

Second pivot

June 17 to June 18

Synthetic data and smart-flow matured in parallel.

Then the story moved below the UI. The Beads epics islandflow-259 and islandflow-zxh split the synthetic and smart-flow plans into a dependency chain: synthetic phase 01 and smart-flow contracts could run in parallel, then fixtures unblocked evidence clustering, labels unblocked scoring, replay unblocked golden tests, and demo profiles unblocked API/UI explainability.

Synthetic market data began with a deterministic spine: seeded profiles, canonical event wrappers, provenance outside payloads, stable ordering, and infra-free tests. It became manifests and fixtures, then scenarios and labels, then replay integration, demo profiles, and hosted controls. In parallel, smart-flow moved from contracts to evidence facts, clustering, score vectors, abstention, API projections, UI explainability, and chart markers.

The recurring choice was to keep truth outside the wrong place. Labels stayed outside emitted market events. Provenance lived by trace ID. Smart-flow projections became canonical, while legacy smart-money streams stayed as fallback. The two epics remain open only because their future calibration phases are intentionally out of the MVP path.

Edge hardening

June 24 to June 25

The hosted UI became the public product surface.

The final implementation stream in this slice was API private edge hardening. The epic islandflow-hnbk started with a read-only baseline of the raw API host, same-origin gaps, bundle exposure, NPM state, and direct port reachability. Then it scrubbed active defaults so docs, scripts, tests, and probes stopped teaching developers and agents to hit concrete production API domains by default.

After that, production browser transport moved to same-origin REST and websocket paths, server-only Next route handlers got an internal API origin, API rate limiting added REST/lookup/websocket buckets and JSON 429 telemetry, the raw API host was closed through durable proxy/deploy state, and the rollout phase proved the final posture live: app-origin REST and websocket checks passed, the raw API host was closed, direct public port 4000 was unreachable, and the production bundle no longer required the raw API origin.

Implementation Milestones

The implementation folders are not side notes. They are the serious phase plans that let me keep large work reviewable: each stream had a Beads epic, phase documents, dependency rules, turn docs or closeout evidence, and a clear definition of what was inside or outside the phase.

Synthetic market data

islandflow-259, phases 01 to 05 complete

Future historical calibration remains open by design.

I made synthetic data deterministic before I let it drive demos.

The synthetic epic started as a planning correction: I did not want demos, fixtures, or smart-flow tests built on ambient runtime randomness. The roadmap in synthetic-market-data/00-roadmap.md made the order explicit. First came a deterministic spine, then manifests and fixtures, then scenarios and labels, then replay integration, then demo and load profiles. Historical calibration stayed future work so the MVP could move without needing paid or brittle historical data first.

Phase 01 created @islandflow/synthetic-market as a reusable package that emits canonical EquityQuote, EquityPrint, OptionNBBO, and OptionPrint event wrappers. I kept provenance keyed by trace id instead of embedding generator metadata into the raw event payloads. Fixed seed and profile inputs produce stable batches, and the generator avoids Date.now() and Math.random() while generating events.

Phase 02 turned that generator into durable artifacts: fixture files, run manifests, replay plans, event hashes, hash-verified loaders, and a Bun CLI. Phase 03 added named scenarios, separate labels, and expected-output sidecars for positive alerts, no-alert cases, suppressions, required evidence, forbidden evidence, and false-positive checks. Phase 04 made replay select synthetic source and run ids, then compare derived smart-flow output signatures. Phase 05 exposed named demo and load profiles so live synthetic emitters could consume fixture-backed scenarios instead of inventing new semantics at runtime.

  • I treated emitted market events as canonical product data, not as synthetic-only pipeline records.
  • I kept hidden labels separate from emitted events so tests could know truth without teaching the runtime impossible facts.
  • I accepted review repairs that tightened ordering around concrete trace_id keys, because replay determinism depends on stable identifiers.
  • I left islandflow-259.6 open for historical calibration because that work tunes realism, while phases 01 to 05 establish repeatable behavior.

Smart-flow

islandflow-zxh, phases 01 to 05 complete

Future calibration remains open after the MVP explanation pipeline.

I rebuilt smart-money into an evidence and abstention system.

The smart-flow roadmap in smart-money/00-roadmap.md chose a careful path: keep the useful working stack, but rebuild the domain model around observations, facts, evidence clusters, cautious hypotheses, confidence, alternatives, abstention, replay evaluation, and user-facing projections. That choice mattered because the old smart-money surface could sound more certain than the data justified.

Phase 01 introduced the canonical contract family in packages/types/src/smart-flow.ts. Phase 02 split evidence work into eligibility facts and deterministic clustering, preserving fact ids and raw evidence refs so later projections could explain themselves. Phase 03 converted clusters into score vectors, alternatives, penalties, confidence, conviction, and explicit abstention reasons. Phase 04 made replay golden tests the acceptance gate, rebuilding smart-flow outputs from raw synthetic fixtures instead of relying on precomputed answers. Phase 05 then exposed the projection through API, websocket, terminal UI, and chart markers.

The important story here is that I stopped treating classification as the only valuable output. A good abstention, with reasons, became a first-class product result. The UI work followed that posture: tables, drawers, command context, priority rows, and chart markers prefer smart-flow projections, while legacy smart-money remains compatibility fallback when no projection is available.

  • I made evidence quality, confidence, conviction, alternatives, and why-not signals explicit instead of burying them under one label.
  • I used synthetic scenarios to validate both positive and negative cases, including unexpected extra alerts and false positives.
  • I split phase 05 into API/websocket surfaces, UI explainability, and chart-marker migration so the presentation layer could be reviewed in manageable slices.
  • I left islandflow-zxh.6 open because calibration should tune a proven pipeline, not compensate for a vague one.

Durable-tapes performance

islandflow-ze79, 11 of 11 complete

Closed after deployed smoke and CDP probe evidence went green.

I measured the request storm before I tried to fix it.

The performance epic was a serious incident stream. The module family was useful, but /durable-tapes was not viable on low-power clients until I could prove request counts, aborts, script time, and task time were bounded. Phase 00 created scripts/probes/durable-tapes-perf.ts and then split it into focused probe modules after review found the initial probe too monolithic. The baseline numbers were intentionally ugly: local hosted-API runs showed thousands of total requests, support lookups, by-trace calls, and aborts while still rendering five panes and 70 rows.

Phase 01 introduced a shared hydration scheduler and caches. The probe moved from roughly eleven thousand total requests to a few hundred, with endpoint aborts eliminated, while CPU remained red for later phases. Phase 02 bounded option-print by-trace lookups with validation, caps, query settings, read timeouts, and JSON error behavior. Phase 03 added a stable terminal store and pane-scoped selectors so route children did not subscribe to the whole terminal context. Phase 04 added keyed incremental live buffers and animation-frame batching, then review repaired snapshot ordering. Phase 05 moved expensive options support hydration and alert evidence decoration server-side through durable row view models. Phase 06 added production smoke checks, latency budgets, fallback flag verification, API-origin routing checks, and deployed CDP probe evidence.

The closeout notes are important because this stream had a real blocker: source guardrails were implemented while deployed production evidence was still red. I did not close the epic until the resolver refreshed native web/API, production smoke passed, and the deployed 30s warmup/180s CDP probe showed zero measured support/evidence requests in the window with acceptable task and script duration. The remaining native ClickHouse systemd health mismatch became a separate follow-up, not hidden in the performance closeout.

  • I made performance observable before optimizing, then carried the same probe through before and after evidence.
  • I reduced the browser storm in stages: cache shared lookups, bound misses, scope subscriptions, update live rows incrementally, then move decorated rows server-side.
  • I required production smoke to prove same-origin routing, durable-row websocket snapshots, options-support latency, and by-trace miss latency.
  • I let thermo-nuclear review force structural repairs, including probe modularization and API durable-row composition extraction.

API private edge

islandflow-hnbk, closed

Closed after live same-origin access worked and raw direct access was shut.

I narrowed the public product surface to the hosted UI.

The API private-edge stream was not a cosmetic deployment change. It changed the posture of the hosted product. The epic described the intended end state: the hosted UI remains public, production browser traffic uses same-origin API and websocket paths, active docs and defaults stop advertising concrete production API domains, and the raw API host becomes private infrastructure rather than a public product surface.

Phase 00 recorded the baseline: raw API host reachability, same-origin route gaps, bundle exposure, NPM route state, relevant env posture, and direct port behavior. Phase 01 scrubbed active defaults and examples so local development falls back to local hosts or explicit placeholders, not the hosted API. Phase 02 switched production browser URL builders to same-origin transport and added a server-only internal API URL for Next route handlers. Phase 03 added env-gated rate limiting with separate REST, lookup, and websocket buckets, forwarded-IP handling, JSON 429 responses, and coarse telemetry. Phase 04 closed the raw public API host through durable NPM/deploy state while preserving app-origin routes and operational server-local access. Phase 05 rolled it out live and proved the final state: blank production NEXT_PUBLIC_API_URL, configured internal API origin, rate limits enabled, app-origin REST and websocket checks passing, raw host closed, public port 4000 unreachable, and no raw origin required by the production bundle.

This stream also refined my agent workflow. The implementation guide required selector, worker, reviewer, and closeout roles, and the reviewer contract said CI stayed with the reviewer until green. It is the clearest example in the repo of Beads, implementation docs, turn docs, review discipline, live verification, and follow-up filing all working together.

  • I separated product publicness from infrastructure reachability: the app is public, the raw API host is not the product.
  • I made same-origin browser transport the default production behavior while preserving explicit local and operator-controlled overrides.
  • I added rate limiting before closing the raw host so the public app-origin paths had abuse bounds.
  • I filed islandflow-od61 for automating native private API bind-host selection instead of folding that future deployment hardening into phase 05.

00 I built the CDP/browser performance probe, budget profile, local/deployed baselines, and later split the probe into focused modules after review. I needed repeatable evidence before changing behavior, and the first baselines captured the request storm clearly.
01 I added shared hydration scheduling, dedupe, in-flight reuse, negative caching, and bounded packet hydration batches. This removed the repeated support/evidence lookup storm and endpoint abort churn.
02 I bounded option-print recent, replay, history, and by-trace reads with validation, caps, storage timeouts, and focused tests. Missing or oversized trace lookups could no longer hang the API or amplify frontend churn.
03 I added a stable terminal store, useTerminalSelector, and memoized durable-tapes pane adapters. Each pane subscribed to the slice it rendered instead of recomputing on unrelated live state.
04 I replaced whole-window live merges with keyed incremental buffers and batched websocket event application. Hot live paths stopped repeatedly sorting and slicing entire windows per event.
05 I moved durable-row composition server-side with typed API/web contracts and explicit hosted fallback behavior. The client received ready-to-render rows instead of performing expensive per-client joins.
06 I added production smoke checks, same-origin fallback checks, CDP budgets, fallback flags, and deployed closeout evidence. I could close the performance epic only after live smoke and browser evidence proved the storm was gone.
Resolver I kept the epic open while deployed production evidence was red, refreshed native web/API, and reran smoke plus the 30s warmup/180s probe. This prevented a source-only victory from masking a deployed failure.

API Private Edge, islandflow-hnbk

Phase What I built Why it mattered
00 I recorded the live edge baseline: raw host behavior, same-origin route gaps, bundle exposure, NPM state, env posture, and direct port reachability. I needed a factual before-state so later closure work could prove what changed.
01 I scrubbed active defaults, examples, tests, probes, and dev fallbacks so they used local defaults or placeholders instead of concrete production API domains. Agents and developers stopped being taught to hit the hosted API by default.
02 I switched production browser REST and websocket builders to same-origin paths and added server-only internal API config for Next route handlers. The hosted UI could talk to the API without advertising the raw API host to browsers.
03 I added API rate limiting with REST, lookup, and websocket buckets, forwarded-IP handling, health exemptions, JSON 429, and telemetry. The public app-origin routes got abuse bounds before direct raw-host access was closed.
04 I closed the raw public API host through durable NPM/deploy state while preserving same-origin app routes and operational local access. The API host changed from public product endpoint to private infrastructure.
05 I rolled out the final live posture, verified app-origin REST and websocket paths, checked raw host closure, checked direct port 4000, and scanned the production bundle. The closeout proved the actual server matched the implementation plan.
Follow-up I filed islandflow-od61 for automating native private API bind-host selection. I kept future deployment hardening visible without widening phase 05 after live acceptance.

Important Twists

These are the turns where the project changed direction or found a hidden constraint.

Docker was default, but native still had to exist.

The deploy docs first clarified Docker-first rollout, then added native units and public edge cutover once operator iteration needed host-level control.

Live data was not enough evidence.

Scroll hold, durable history, and ClickHouse alert context all came from the same realization: the user investigates old rows while new rows keep arriving.

Forgejo became the center of gravity.

GitHub Pages stayed useful for published docs, but implementation workflow, PRs, pushes, CI, and Beads sync all moved toward Forgejo as canonical.

Mock routes were not throwaway screens.

The mock route sequence became the design decision engine, and mock9 through mock12 eventually defined the production terminal routes.

Labels and provenance were kept out of event payloads.

Synthetic data became credible because generation metadata and ground truth remained traceable without polluting canonical market events.

Smart-money became smart-flow projection.

The old stream stayed compatible, but the product started preferring canonical explainability projections for tables, drawers, priorities, and chart markers.

terminal.tsx became a facade.

The refactor did not remove behavior first. It preserved imports and routes while extracting helpers, hooks, state, shell, UI, and only then retired unreachable code.

Reusable tapes created a performance problem worth measuring.

The durable tape modules were the right product boundary, then the performance epic proved the browser needed request budgets, pane-scoped subscriptions, and server-composed rows.

The API became infrastructure, not a public product surface.

The private-edge stream moved production browser traffic to same-origin paths, rate-limited public access, and closed direct raw-host usage after live verification.
