Fix Forgejo Security Image Issues

Addressed the four open Forgejo security issues by pinning Redis and Bun deployment images to patched immutable manifests.

Forgejo #19 Forgejo #44 Forgejo #45 Forgejo #46 2026-06-19 12:45

Summary

The Redis runtime image now stays on the patched Redis OSS 7.2 line with `redis:7.2.14-bookworm@sha256:747f1f4078d700806f83d427550a9d8c4d89dd3bf6491449c748c83ad285036e`. The Bun-based deployment Dockerfiles now use `oven/bun:1.3.14@sha256:e10577f0db68676a7024391c6e5cb4b879ebd17188ab750cf10024a6d700e5c4` instead of `oven/bun:1.3.11`.

Changes Made

  • Changed the VPS Redis service in `deployment/docker/docker-compose.yml` from floating `redis:7.2` to the patched digest-pinned Redis image.
  • Changed the local root `docker-compose.yml` Redis service to the same patched image so local infrastructure scans do not keep reporting the old tag.
  • Changed `deployment/docker/Dockerfile.service` to the patched digest-pinned Bun image.
  • Changed both build and runtime stages in `deployment/docker/Dockerfile.web` to the same patched Bun image.
  • Changed `deployment/docker/Dockerfile.ingest-options` to the same patched Bun image.

Context

Forgejo issues #44, #45, and #46 reported Redis CVEs against `redis:7.2` in the deployed runtime image. Forgejo issue #19 reported CVE-2026-45447 against OpenSSL packages inherited from `oven/bun:1.3.11` in the service, web, and ingest-options Docker images.

The Redis update stays on the existing 7.2 minor line and pins the patched `7.2.14-bookworm` multi-arch index. The Bun update moves to the latest verified stable 1.3 image available during the task and pins its multi-arch index.

Important Implementation Details

  • `docker buildx imagetools inspect redis:7.2.14-bookworm` confirmed the Redis multi-arch digest `sha256:747f1f4078d700806f83d427550a9d8c4d89dd3bf6491449c748c83ad285036e`.
  • `docker buildx imagetools inspect oven/bun:1.3.14` confirmed the Bun multi-arch digest `sha256:e10577f0db68676a7024391c6e5cb4b879ebd17188ab750cf10024a6d700e5c4`.
  • The readable tags remain next to the digest pins, which keeps review clarity while preventing future rebuilds from silently floating to a different manifest.
  • The deployment compose file requires `deployment/docker/.env`; validation used either `--no-env-resolution` or a temporary copy of `.env.example`, removed immediately afterward.

Expected Impact for End-Users

Rebuilt deployment images should stop inheriting the vulnerable Redis and Bun base-image surfaces identified by Forgejo. Runtime behavior is expected to remain unchanged: Redis stays on the same minor release line, and Bun stays on the same 1.3 line while receiving patched base packages.

Validation

  • Passed: `rg` confirmed the old `oven/bun:1.3.11` and unpinned `image: redis:7.2` references are gone from the deployment files.
  • Passed: `docker buildx imagetools inspect redis:7.2.14-bookworm` confirmed the Redis digest pin.
  • Passed: `docker buildx imagetools inspect oven/bun:1.3.14` confirmed the Bun digest pin.
  • Passed: `docker compose -f docker-compose.yml config --quiet` parsed the root local compose file.
  • Passed: `docker compose -f deployment/docker/docker-compose.yml config --quiet --no-env-resolution` parsed the deployment compose file without requiring a local `.env`.
  • Passed: `git diff --check` reported no whitespace errors.

Issues, Limitations, and Mitigations

  • Not run: image pulls, in-container package checks, and vulnerability rescans, because the local Docker API socket was unavailable and `trivy` was not installed.
  • Mitigation: both replacement image manifests were verified through Docker registry metadata, and compose parsing verified the updated references are accepted by Docker Compose.
  • Follow-up validation: after Docker or OrbStack is available, rebuild the deployment images and rescan them with the same scanner that opened the Forgejo issues.
