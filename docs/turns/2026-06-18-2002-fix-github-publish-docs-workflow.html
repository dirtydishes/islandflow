Fix GitHub Publish Docs Workflow

Made the GitHub docs publisher deterministic, removed the missing cross-repo secret dependency, regenerated the stale docs index, and prepared the missed documentation payload for Pages.

120 docs indexed after backfill
66 files observed in the current GitHub Pages mirror before backfill
120 files in the locally prepared static payload, including docs/index.html

Summary

The Publish Docs workflow had a stale generated index and the GitHub source repository was behind Forgejo, so newer documentation never reached GitHub Pages. The first fixed run proved the build path worked but failed because DOCS_PAGES_TOKEN was not configured. This change makes the workflow explicitly install Node, checks out full git history for stable commit dates, publishes with the built-in GITHUB_TOKEN, and regenerates the index to include all tracked docs.

Changes Made

  • Added full-history checkout and Node 22 setup to .github/workflows/docs-pages.yml.
  • Changed the workflow to publish to the same repository's gh-pages branch with the built-in checkout token instead of requiring a missing cross-repo DOCS_PAGES_TOKEN.
  • Changed scripts/generate-docs-index.mjs to read tracked docs from git ls-files docs.
  • Changed docs timestamps to use each file's last commit time instead of checkout filesystem mtime.
  • Regenerated docs/index.html so it lists 120 docs, including implementation and research docs that were absent from the stale index.

Context

The local Forgejo checkout was at eb46225, while the GitHub mirror's main ref was still at 214283a. The existing user-site Pages repository had only 66 files under islandflow/docs, while this repo currently has 120 tracked docs excluding the index. That mismatch is the backfill gap.

Important Implementation Details

  • The generator now depends on git being available, which is already true in GitHub-hosted Actions runners and local repo worktrees.
  • fetch-depth: 0 is required because shallow checkout history would not reliably provide per-file last-commit dates.
  • New docs without a committed history entry use filesystem mtime locally; after CI checkout, committed docs use the git commit date.
  • The generated index excludes itself, then the workflow copies all of docs/ into site/docs/ for publication.
  • The generator template was tightened so regenerated HTML does not introduce trailing whitespace.

Issues, Limitations, and Mitigations

  • The local shell did not have gh, node, or bun on PATH, so validation used the bundled Node runtime and git commands.
  • bd was also unavailable on PATH, so Beads could not be updated with the normal CLI during this run.
  • The GitHub source mirror was behind Forgejo at the start of this fix; final handoff should confirm both remotes have the completed docs-publish commits.
