Islandflow Turn Record

Smart-flow replay evaluation now has deterministic golden tests.

2026-06-18 14:27 Beads: islandflow-zxh.4 Branch: lavender/replay-golden-tests

Summary

Added a pure compute-side replay evaluator that rebuilds smart-flow evidence, clusters, hypotheses, and compact signatures from raw synthetic market events. Added infra-free golden tests for positive alert, unexpected extra alert, no-alert abstention, false-positive detection, and noisy event-context replay. A review pass then hardened the golden comparator so an expected alert cannot hide additional emitted alert signatures.

Changes Made

  • Added services/compute/src/smart-flow-replay-evaluation.ts, which turns raw synthetic option events into flow packets, recomputes smart-flow evidence and hypotheses, and compares signatures to expected manifests.
  • Added services/compute/tests/smart-flow-replay-evaluation.test.ts with five deterministic golden cases: positive, unexpected extra alert, abstention, false-positive, and noisy event context.
  • Hardened expected-manifest comparison so emitted alerts are consumed against expected class/direction signatures exactly once. Extra signatures are now reported as unexpected_class, unexpected_direction, or false_positive instead of passing beside a valid alert.
  • Added @islandflow/synthetic-market as a compute workspace dependency so the evaluator can use fixture types and stable hashing.

Context

Smart-flow phase 04 asks replay to become the acceptance gate for evidence and hypothesis behavior. The important constraint is that replay starts from raw synthetic fixture events, not pre-derived events, so regressions in evidence extraction or hypothesis scoring remain visible.

Important Implementation Details

  • Raw events are ordered by ts, ingest_ts, seq, and trace_id before flow packets are built.
  • Signatures intentionally keep only stable review fields: counts, candidate eligibility, cluster feature summaries, hypothesis type/direction, confidence, abstention reasons, and penalty kinds.
  • Positive golden manifests now reject extra emitted alert signatures, including wrong-direction alerts emitted from other raw replay clusters.
  • No service containers are required. The tests use generated synthetic fixtures in memory and compare expected-output manifests directly.
  • No UI explainability, historical calibration, ClickHouse, NATS, Redis, or large generated dumps were added.

Issues, Limitations, and Mitigations

  • The evaluator is intentionally fixture-backed and infra-free. It does not try to validate ClickHouse, NATS, Redis, or Docker-backed replay paths.
  • The signature is compact by design. It is meant for stable acceptance checks, not exhaustive payload snapshots.
  • Full repository Biome remains blocked by unrelated import-order diagnostics; touched-file Biome and full typecheck both pass.
  • Review found and fixed a comparator gap: an expected positive alert could previously pass while another emitted alert signature remained unreported.
