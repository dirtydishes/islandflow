Islandflow phase 05b

Smart-flow explainability in the terminal UI

Implemented the UI side of the smart-flow explainability phase: hypotheses now foreground evidence quality, confidence versus conviction, why-not context, abstention, alternatives, and inspectable references without claiming participant identity.

Issue islandflow-zxh.5.2 Branch lavender/smart-flow-ui-explainability UI bounded Infra-free validation

Summary

The terminal signals surface now consumes the canonical smart-flow explainability projection feed from the phase 05a API/WS work and presents it as cautious, evidence-backed flow hypotheses. The old smart-money stream remains as a compatibility fallback, but the primary table, drawer, command ticker context, and priority rows now prefer smart-flow projections.

Changes Made

Smart-flow live feed

Added smart-flow to the terminal live manifest, history loading, hot-window state, replay tape, and older-window loading path.

Explainability table

Reworked the former classifier pane into a Flow Hypotheses table with hypothesis, direction, confidence, conviction, evidence quality, and why-not columns.

Inspectable drawer

Added a smart-flow drawer that shows the hypothesis read, confidence versus conviction, abstention, alternatives, evidence refs, hydrated packet/print evidence, and version trace.

Cautious copy

Rephrased visible labels away from overconfident smart-money language and kept legacy profile UI framed as compatibility context.

Context

This child issue is the PR-sized UI portion of phase 05. It intentionally builds on the API and websocket contracts shipped by PR 40 instead of reimplementing those surfaces. The scope was also constrained to explainability presentation; scoring policy, calibration, participant identity, and synthetic foundations were left alone.

Important Implementation Details

  • Smart-flow projections are subscribed before legacy smart-money events when smart-money features are enabled, so the UI can prefer canonical projections while keeping existing compatibility consumers alive.
  • Evidence refs are merged from projection refs, evidence refs, and hypothesis refs, then split into flowpacket: references and option-print references for hydration through existing fetchers.
  • Why-not copy is deterministic: abstention reasons win, then evidence penalties, then the strongest alternative hypothesis, then a neutral "no active guard" state.
  • Dashboard command context uses smart-flow rows first and falls back to legacy smart-money rows only when no smart-flow projections are present.

Expected Impact for End-Users

A user looking at the terminal should see smart-flow as an explainable hypothesis stream instead of a black-box or overconfident "smart money" call. They can inspect why a read exists, why it may be weak, what alternatives were considered, and which packets or prints support the displayed hypothesis.

Followup Changes

Review follow-up tightened the live smart-flow path without expanding phase scope. Empty reconnect snapshots now retain existing smart-flow rows like the other history-backed live channels, selected smart-flow drawers are cleared when switching live/replay mode, and selected smart-flow packet/print refs are protected from pinned-evidence pruning while the drawer is open.

Abstained projections now render their row and drawer direction as neutral abstained language instead of showing the underlying bullish/bearish direction as if the system had made a directional call.
