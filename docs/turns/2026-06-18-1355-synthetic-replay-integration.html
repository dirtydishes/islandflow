Synthetic Market-Data Phase 04: Replay Integration

Beads issue islandflow-259.4. Created June 18, 2026 at 13:55.

replay synthetic fixtures manifest comparison infra-free tests
Summary

Added opt-in fixture-backed synthetic replay support to the replay service. Synthetic runs can now be selected by source and run ID, replayed in manifest order, and checked against expected smart-flow output signatures without Docker, ClickHouse, NATS, or Redis for the fast test path.

Changes Made

  • Added REPLAY_SOURCE=synthetic_fixture mode to services/replay, while preserving the default ClickHouse replay path.
  • Added synthetic fixture selectors for source_id and run_id, with strict validation against fixture provenance.
  • Added a replay helper that orders synthetic market events by ts, ingest_ts, seq, and stable event_id.
  • Added manifest signature comparison for smart-flow expected-output sidecars.
  • Updated the synthetic manifest replay ordering label from trace_id to event_id to match the phase wording while still using trace IDs as the current stable event IDs.
  • Documented new replay environment variables in README.md.

Context

Phase 04 is deliberately narrow: replay should consume deterministic synthetic runs, preserve the same ordering semantics used by the generator and replay service, and make expected-output comparison available. This change does not add demo controls, load profiles, historical calibration, or smart-flow scoring policy changes.

Important Implementation Details

Default path unchangedThe replay service still uses ClickHouse unless REPLAY_SOURCE=synthetic_fixture is set.
Selector safetyFixture replay rejects source IDs other than synthetic_market and run IDs that do not match the loaded manifest.
Stable orderingReplay order is checked against the manifest trace/event ID list after sorting by event time, ingest time, sequence, and stable event ID.
Signature comparisonExpected smart-flow output sidecars are hashed and compared to the manifest hash rather than comparing noisy full payloads everywhere.

        event_id: getEventStableId(next.event)

Expected Impact for End-Users

Researchers and developers can replay named synthetic fixtures through the same replay publisher semantics used for live/raw streams. That makes synthetic runs useful as deterministic acceptance fixtures before future smart-flow golden tests and UI demos depend on them.

Issues, Limitations, and Mitigations

  • Fixture replay is opt-in and still publishes through NATS, so running the service mode requires NATS. The fast validation helpers and tests remain infra-free.
  • Materialized ClickHouse rows do not yet carry full synthetic provenance columns, so strict source/run selection is implemented for fixtures. Existing trace-prefix filtering remains the materialized-row selector available today.
  • The stable event ID currently maps to trace_id. This is explicit in the helper so a future canonical event ID field can replace it in one place.
