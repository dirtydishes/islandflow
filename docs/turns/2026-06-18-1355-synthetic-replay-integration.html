Synthetic Market-Data Phase 04: Replay Integration

Beads issue islandflow-259.4. Created June 18, 2026 at 13:55 EDT. Updated during PR review at 14:05 EDT.

replay synthetic fixtures manifest comparison infra-free tests
Summary

Added opt-in fixture-backed synthetic replay support to the replay service. Synthetic runs can be selected by source and run ID, replayed in manifest order, and checked against expected smart-flow output signatures. The PR review pass also corrected the manifest ordering key back to trace_id, which is the concrete stable identifier present on the current market events.

Changes Made

  • Added REPLAY_SOURCE=synthetic_fixture mode to services/replay, while preserving the default ClickHouse replay path.
  • Added synthetic fixture selectors for source_id and run_id, with strict validation against fixture provenance before replay starts.
  • Added a replay helper that orders synthetic market events by ts, ingest_ts, seq, and stable trace_id.
  • Added manifest signature comparison for smart-flow expected-output sidecars.
  • Added regression coverage to keep the fixture replay ordering key aligned with the actual event schema.
  • Documented new replay environment variables in README.md.

Context

Phase 04 is deliberately narrow: replay should consume deterministic synthetic runs, preserve manifest ordering, and make expected-output comparison available before downstream smart-flow golden tests depend on replay semantics. This change does not add demo controls, load profiles, historical calibration, or smart-flow scoring policy changes.

Important Implementation Details

Default path unchangedThe replay service still uses ClickHouse unless REPLAY_SOURCE=synthetic_fixture is set.
Selector safetyFixture replay rejects source IDs other than synthetic_market and run IDs that do not match the loaded manifest.
Stable orderingReplay order is checked against the manifest trace_ids list after sorting by event time, ingest time, sequence, and trace ID.
Signature comparisonExpected smart-flow output sidecars are hashed and compared to the manifest hash rather than comparing noisy full payloads everywhere.

Users and maintainers get a fast, infra-free way to verify replay ordering and expected-output signatures while keeping the production replay default unchanged.

Issues, Limitations, and Mitigations

  • Fixture replay is opt-in and still publishes through NATS in service mode. The replay-plan helpers and tests remain infra-free, which keeps default validation fast.
  • Materialized ClickHouse rows do not yet carry strict synthetic provenance columns, so source/run validation is enforced on fixture-backed replay. Materialized synthetic replay should stay optional until provenance columns exist.
  • The current stable market-event identifier is trace_id. If a future canonical event_id field is added to market events, the manifest schema and replay helpers should be migrated together with a versioned compatibility path.
