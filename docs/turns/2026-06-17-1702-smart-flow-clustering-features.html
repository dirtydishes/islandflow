Repository Turn Document

Smart-flow evidence clustering and traceable feature vectors

Implemented Beads issue islandflow-zxh.2.2, the phase 02b slice that groups explicit evidence candidates into deterministic clusters and derives feature values with source lineage.

Tests passed Phase 02b No scoring policy

Summary

This turn adds a smart-flow clustering layer on top of the phase 02a evidence facts. Eligible evidence candidates are grouped into deterministic time windows, rejected candidates are kept out by default with their IDs reported separately, and every cluster feature carries fact IDs plus raw evidence refs.

Changes Made

  • Extended smart-flow contracts with FlowFeatureBasis and FlowTraceableFeature so cluster feature values can declare whether they are measured facts, derived metrics, or inferred structure.
  • Added feature_details to FlowEvidenceCluster next to the existing feature_summary, preserving compact values and per-feature lineage.
  • Added services/compute/src/smart-flow-clusters.ts, which builds deterministic cluster IDs and windows from evidence candidates.
  • Aggregated measured features such as premium, size, quote coverage, stale ratio, aggression, and spread values from evidence facts.
  • Added derived and inferred features for candidate counts, packet counts, member counts, eligibility status, and structure context without introducing hypothesis scoring.
  • Added infra-free tests for key stability, rejected-candidate separation, feature aggregation, and traceability.

Context

The selected issue was islandflow-zxh.2.2, the PR-sized child under smart-flow phase 02. The implementation README places this phase after synthetic manifests and after phase 02a eligibility facts, and before synthetic scenario labels. The phase document limits this work to evidence clustering and traceable features, so final hypothesis scoring, calibration, API explainability, UI explainability, and storage replacement remain out of scope.

Important Implementation Details

Deterministic Windows

Cluster IDs use cluster:<underlying>:<start>:<end>. Inputs are sorted by observed timestamp and candidate ID, so cluster keys and summaries stay stable regardless of input order.

Lineage

Each feature_details entry stores fact_ids and evidence_refs. The summary remains plain values, but the detailed layer makes the audit trail explicit.

Scope Boundary

The builder does not score hypotheses or claim participant intent. It only organizes evidence and computes cautious feature values for later phases to consume.

Issues, Limitations, and Mitigations

  • The clustering helper is not wired into the live compute service yet. That keeps this phase slice reviewable and avoids changing runtime signal behavior before scoring policy exists.
  • Rejected candidates are excluded from clusters by default, but the builder returns their candidate IDs and has an opt-in includeRejectedCandidates configuration for future diagnostics.
  • Feature aggregation is intentionally simple and auditable. Historical calibration, baseline tuning, and confidence scoring remain future phase work.
  • The full Biome repo check still fails outside this change because unrelated import-order cleanup is already tracked separately.
