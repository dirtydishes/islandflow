Islandflow Turn Record

Smart-flow explainability is now exposed through API and websocket surfaces.

2026-06-18 19:32 Beads: islandflow-zxh.5.1 Branch: lavender/smart-flow-evidence-api-websocket

Summary

Added a canonical smart-flow explainability projection and exposed it through recent, history, replay, one-channel websocket, and /ws/live subscription surfaces. The payload includes version metadata, refs, evidence handles, hypothesis and insight objects, alternatives, and abstention details.

Existing raw smart-money routes and websocket feeds remain available for compatibility. They now also feed the new smart-flow projection path, but they were not changed into canonical contracts in this PR.

Changes Made

  • Added SmartFlowExplainabilityProjection to packages/types/src/smart-flow.ts, including refs, evidence, hypothesis, insight, abstention, alternatives, and version fields.
  • Added smart-flow as a typed live channel in packages/types/src/live.ts and the API live-state manager.
  • Added services/api/src/smart-flow.ts to project persisted legacy smart-money events into compatibility-tagged smart-flow explainability payloads.
  • Added REST surfaces at /flow/smart-flow, /history/smart-flow, and /replay/smart-flow, plus /ws/smart-flow and /ws/live support.
  • Added a smart_flow field to the options-support lookup response so existing packet context can carry the new explainability projection alongside the legacy smart_money field.
  • Added focused contract, API projection, and live-state tests for the new surface.

Context

Phase 05a is intentionally API/WS-only. Earlier smart-flow phases already created the evidence, hypothesis, abstention, and replay-golden foundations. This turn exposes those semantics at the boundary without changing scoring policy, calibration, participant identity, synthetic foundations, or phase 05b UI explainability surfaces.

Important Implementation Details

Area Detail
Canonical payload SmartFlowExplainabilityProjection duplicates the handles clients need at top level while preserving the nested hypothesis and insight contracts.
Compatibility New payloads projected from legacy events carry compatibility_only: true, legacy_event_id, and legacy_channel: "smart-money".
Live delivery The smart-money NATS pump now broadcasts raw smart-money events to existing subscribers and projected smart-flow explainability payloads to the new channel.
Replay and history Smart-flow cursor semantics remain tied to the underlying validated event source_ts and seq, matching existing replay/history behavior.

Expected Impact for End-Users

API and websocket clients can now consume smart-flow explanation payloads directly: evidence refs, hypothesis details, insight copy, abstention state, and alternative explanations arrive in a stable versioned contract. Current UI clients that still read raw smart-money feeds should continue to work unchanged.

Issues, Limitations, and Mitigations

  • Projected source: the API projection currently derives from persisted legacy smart-money events because this phase did not add a canonical hypothesis storage table. Payloads identify this with compatibility metadata.
  • Startup cost: live hydration can query the smart-money table once for raw smart-money and once for smart-flow explainability. Limits are small, and this keeps compatibility simple for phase 05a.
  • UI migration: no phase 05b UI explainability surface was implemented. The new API/WS contracts are ready for that follow-up.
