Summary

Reworked the repo-root deploy helper so running ./deploy opens a guarded interactive prompt, while existing scripted deploy commands still work. The script now supports choosing Docker or native runtime, main/current/another branch, and exact app pieces such as api, web, compute, and ingest workers.

Changes Made

  • Added interactive deploy prompts for runtime, git target, deploy pieces, fast mode, build skipping, Docker force recreate, and final confirmation.
  • Added ./deploy branch <name> for deploying a named local or remote branch without switching the current worktree.
  • Added --pieces and repeatable --piece flags for explicit multi-piece rollouts.
  • Preserved legacy scope flags such as --web-only, --api-only, --services-only, and --workers-only.
  • Documented guided and scripted rollout examples in Docker and native deployment docs.
  • Added @pierre/theme so Diffs SSR can render turn-document snippets reliably.

Context

The previous helper already handled remote execution, Docker/native rollout paths, prechecks, health checks, timing summaries, and public verification. The missing operator experience was a single safer front door for choosing deployment shape without remembering every flag.

Important Implementation Details

  • Interactive mode is TTY-only. Non-terminal invocations with no arguments print usage instead of blocking automation.
  • All remains the default. In guided mode, pressing enter at the pieces prompt selects every core app piece.
  • Fast mode keeps old semantics. With all pieces selected, Docker fast mode narrows to services and native fast mode narrows to workers.
  • Another branch is non-invasive. Named branch deploys push a matching local branch if present, otherwise use the fetched remote branch.
  • Native edge guardrails remain. Native deploys that touch web/API still require DEPLOY_NATIVE_EDGE_READY=1.

Expected Impact for End-Users

Operators can now run one command, answer a short set of prompts, and deploy the right runtime, branch, and app pieces with a final confirmation before anything remote changes. Scripted automation remains compatible with the previous flags, and partial rollouts can be more precise than the old grouped scopes.

Issues, Limitations, and Mitigations

  • The guided prompt cannot be fully smoke-tested through a pipe because it intentionally requires a TTY. The usage path and parser failure paths were validated non-interactively.
  • The current worktree is detached, so current branch is correctly unavailable in guided mode until the checkout is on a branch.
  • Named branch deploys still depend on the server checkout accepting a normal git switch and git pull --ff-only.

Follow-up Work

No required follow-up remains for this request. A later enhancement could add a dedicated --dry-run mode that prints the selected rollout plan without running prechecks or remote commands.
