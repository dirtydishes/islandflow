What This Plan Is Really Building

Islandflow needs two capabilities to mature together. First, it needs a controlled market-data world: fake options and equity tape that is realistic enough to test alerts, but deterministic enough to replay. Second, it needs a cautious smart-flow engine that can explain whether a market event is useful evidence, ambiguous noise, or a reason to abstain.

Synthetic market data

This is not random demo filler. It is a seeded generator that emits normal market events like OptionPrint, OptionNBBO, EquityPrint, and EquityQuote. Same seed and profile means same output.

Smart-flow detection

This replaces overconfident "smart money" claims with evidence-backed hypotheses. It keeps direct facts, evidence quality, signal strength, penalties, alternatives, confidence, conviction, and abstention separate.

Replay as the proof loop

Replay is the bridge between both tracks. It reruns raw synthetic streams and checks whether the derived evidence and hypotheses match compact expected signatures.

Two Tracks, One Pipeline

The synthetic track creates the market conditions. The smart-flow track interprets those conditions. The phases alternate because each side needs the other side's vocabulary or artifacts before the next step is useful.

Research Rules That Shape The Implementation

The research docs are background, but they explain the guardrails behind the phase order. These are the rules the implementation should keep visible.

  • Synthetic data should be messy on purpose, not random. Real tape has clustered arrivals, lumpy sizes, variable spreads, stale quotes, odd lots, late/corrected prints, off-exchange reports, sparse options chains, 0DTE behavior, IV/skew structure, and regime changes. The generator needs enough of that mess to test false positives.
  • Background runs should usually fail high-confidence alert logic. A useful background stream is not empty. It has ambiguous bursts, noisy context, weak confirmations, and ordinary traps. High-confidence alerts should be rare outside deliberate scenarios; abstentions should be common enough to prove the model is not overclaiming.
  • Labels must stay separate from emitted market events. Ground truth belongs in label records and expected-output manifests. If labels leak into canonical event payloads, production-like paths can accidentally cheat.
  • Smart-flow is a hypothesis engine, not a participant-identity engine. Public data can support evidence-backed directional, volatility, hedge/reactive, or event-flow candidates. It cannot directly prove the hidden trader or parent order. The product should show confidence and alternatives, not certainty theater.
  • Replay must run in event time and preserve ordering. Validation should not depend on when the app happened to receive a message. Stable run IDs, event IDs, event-time ordering, and compact output signatures are the backbone of repeatable tests.
  • Historical calibration is valuable, but not an MVP dependency. Later calibration can fit arrival curves, spread states, size mixtures, venue shares, option-chain weights, confidence thresholds, and abstention behavior. It should layer onto deterministic foundations, not replace them.

The Smart-Flow Scoring Shape

The research pushes away from one big "smart money score." The implementation should keep the pieces separate so the system can explain why a signal is strong, weak, or not worth calling. 

evidence_quality
  = quote freshness
  + spread and liquidity quality
  + condition-code eligibility
  + timestamp and reference-data confidence

signal_strength
  = contract-relative premium/size anomaly
  + volume versus prior open interest
  + sweep or burst persistence
  + IV/skew and underlying confirmation

false_positive_penalty
  = stale quote
  + wide spread
  + complex spread or hedge likelihood
  + 0DTE/event noise
  + late/corrected/off-hours prints

hypothesis
  = evidence quality + matching signal strength - penalties

if evidence is weak, explanations conflict, or the runner-up hypothesis is too close:
  emit abstention with reasons instead of a confident insight

Terms Worth Keeping Straight

These terms are intentionally not hidden. They are the practical handles someone needs when reading the phase docs or reviewing implementation work.

Deterministic

Fixed inputs produce fixed outputs. In this repo that means seeds, profiles, generator version, event ordering, and parameter snapshots are pinned so a test can reproduce the same run later.

Canonical market events

Synthetic data should use the same raw pipeline contracts as real data: option prints, option quotes, equity prints, and equity quotes. Do not make a parallel synthetic-only business schema for the main pipeline.

Manifest

A durable description of a generated run: seed bundle, profile identity, generator version, parameter hash, event hashes, replay ordering, and expected derived outputs.

Ground-truth label

A separate record that says what a synthetic scenario is intended to mean. It can name expected class, direction, confidence band, required evidence, forbidden evidence, and penalties. It must not be embedded in the market event.

Evidence cluster

A traceable group of observations and evidence facts. It can include raw refs, quote context, eligibility decisions, feature values, and evidence quality before any final hypothesis is scored.

Hypothesis

A model or policy output over evidence. It is not a direct fact about a hidden participant. It should carry scores, alternatives, policy/model versions, and abstention where needed.

Confidence versus conviction

Conviction is about signal strength. Confidence is about how cleanly the evidence supports one hypothesis over alternatives. A large print can be high-conviction but low-confidence when context is messy.

Abstention

A first-class output that says the system should not make a strong claim. Good abstentions name reasons such as stale quotes, wide spreads, event noise, complex spread likelihood, or conflicting evidence.

Golden test

A compact expected-output check for replayed behavior. It should compare stable signatures and meaningful fields rather than huge payload dumps that reviewers stop reading.

What Not To Jump To Yet

These are tempting because they feel product-visible. The plan delays them because they become misleading unless replay and evidence semantics are already solid.

Do not build UI certainty first

API and UI explainability should wait for replay-validated hypotheses. Otherwise the product may teach users to trust old "smart money" certainty with new labels.

Do not make demos random again

Demo controls should select named deterministic runs. Hidden wall-clock randomness makes impressive demos but weak tests.

Do not require historical data for MVP

Historical calibration belongs after the deterministic MVP chain. Use it later to improve parameters, not to block the foundation.
