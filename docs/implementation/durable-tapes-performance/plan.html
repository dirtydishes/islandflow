Durable-Tapes Performance Hardening

A staged repair plan for making /durable-tapes usable on low-power clients by stopping frontend request storms, bounding lookup endpoints, reducing React invalidation, replacing hot merge/sort windows, and moving row decoration server-side.

Epic: islandflow-ze79 Source bug: islandflow-ba9q Index: IMPLEMENT.md Loop: selector -> worker -> reviewer -> closeout
Diagnosis Phases Agent Loop PR Guidance Subagents Quality Gates

Diagnosis

What is melting the browser

The websocket volume is not large enough by itself to explain the meltdown. The expensive part is multiplier work after each live update: repeated scans, repeated support/evidence fetches, aborted requests, broad React recomputation, and full-window live row merging.

What must change

The client should stop asking the same support questions repeatedly, the API must return quickly on lookup misses, panes should subscribe to smaller slices, hot buffers should update incrementally, and the server should eventually send rows that are already decorated enough to render.

Agent Loop

Selector

Read IMPLEMENT.md, run bd prime, bd ready, and bd show. Report the next ready phase, dependency state, linked phase doc, and safe parallelism. Do not select from memory.

Worker

Receive the full phase doc, issue ID, quality gates, branch/PR posture, turn-doc location, and orchestrator thread ID. Keep the PR phase-bounded and call back once when PR-ready or blocked.

Reviewer

Review with findings first. Repair in scope when safe, update the existing phase turn doc, verify CI after any repair commit, and call back only when the review state is resolved.

Closeout

Merge only after reviewer callback and green CI. Sync Beads, push Forgejo, verify clean status, then rerun selector until the epic is complete.

PR Guidance

  • Keep one active implementation PR at a time unless Phase 01 and Phase 02 are intentionally parallel after Phase 00.
  • Do not create branches unless the orchestrator explicitly assigns one.
  • Keep Phase 01 focused on frontend scheduler/cache behavior.
  • Keep Phase 02 focused on API/storage lookup bounds.
  • Do not mix server-composed row models into pane-state or buffer phases.
  • Split Phase 05 into child issues if the row-model contract becomes too large for one review.
  • File follow-up Beads issues for adjacent cleanup instead of widening scope.

Subagent Delegation

Good tasks

  • Inventory support/evidence lookup call sites.
  • Review storage tests and API miss behavior.
  • Run the perf probe and summarize deltas.
  • Audit pane subscription invalidation.
  • Compare row-model contract alternatives.

Do not delegate

  • Public scheduler interface decisions.
  • Server row-model contract decisions.
  • Beads dependency changes.
  • Branch, PR, commit, push, merge, or closeout ownership.
  • Reading or interpreting skill instructions.

Quality Gates

Use phase-specific gates first. Common gates are:

bun test apps/web/app/terminal.test.ts apps/web/app/routes.test.ts
bun test apps/web/features/terminal apps/web/features/durable-tape
