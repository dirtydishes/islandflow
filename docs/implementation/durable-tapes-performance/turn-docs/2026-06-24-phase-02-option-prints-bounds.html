Durable Tapes Performance Phase 02: Option Prints Bounds

Bounded option-print storage queries and API lookup validation so recent option-print requests and trace-id misses return JSON errors or empty data quickly instead of holding API workers until the proxy times out.

Branchlavender/ze79-phase-02-option-prints-bounds
Beadsislandflow-ze79.3
ScopeAPI/storage only
Timestamp2026-06-24

Summary

  • Added bounded ClickHouse query settings and client-side request/body read timeouts for option-print recent, replay, history, and by-trace storage reads.
  • Validates /option-prints/by-trace request batches at the API boundary: empty requests return [], oversized unique batches return 400, overlong or control-character trace IDs return 400, and storage failures return bounded 503 JSON.
  • Dedupes and caps storage trace lookups defensively even when called outside the public route, while filtering blank and overlong IDs before building SQL.
  • Adds a ClickHouse trace-id bloom skipping index to the option-prints DDL and migration path so new table parts have a trace-optimized lookup aid without adding a new frontend contract.
  • Adds lightweight api.option_prints.by_trace_ms timing metrics tagged as hit, miss, invalid, or error.

Diagnosis

The table definition orders option_prints by (ts, option_contract_id), while /option-prints/by-trace looked up by trace_id IN (...). That means a miss has no primary-key help. The public route failure on /prints/options?view=signal&limit=1 is the adjacent recent-query side of the same storage family: the route had no query timeout or request timeout, so a slow ClickHouse read could outlive the public proxy.

This PR intentionally keeps the fix in API/storage. It does not change durable-tapes rendering, hydration scheduling, or any frontend retry behavior.

  • infra unavailable local API startup was not attempted because Docker/OrbStack was unavailable: docker ps could not connect to the daemon socket.
  • deployed recovery pending The public API route check targets the live deployment. This branch can only prove live public recovery after the API containing these bounds is deployed.

Known Risks

  • The trace-id skipping index is added for new table parts. Existing parts may still need ClickHouse materialization or a dedicated projection/table if trace misses remain slow after deployment, but the request and query timeouts now bound the failure mode. Follow-up: islandflow-ze79.8.
  • If the production ClickHouse version rejects ADD INDEX IF NOT EXISTS, the migration will need a version-specific follow-up. The local tests verify generated DDL, not a live migration against the production server.
